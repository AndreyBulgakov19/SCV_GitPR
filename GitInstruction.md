#  Работа с GIT.


## 1. Проверка наличия установленного GIT

В Терминале выполнить команду *__git version__* (
    ```git version```) Если Git установлен, то появится сообщение с версией программы
``` Bash
git version 2.42.0.windows.2 
```

иначе будет сообщение об ошибке.

## 2. Установка GIT.

   Загружаем последнюю версию Git с [сайта](https://git-scm.com/downloads) 
Устанавливаем с настройками по умолчанию.

## 3. Настройка GIT.

При первом использовании GIT необходимо 
представиться. Для этого нужно ввести в терминале две команды.
```Bash
git config --global user.name "Ваше имя английскими буквами"
git config --global user.email "Ваша почта@example.com"
```

## 4. Инициализация репозитория.

Чтобы создать новый репозиторий, нам нужно открыть терминал, зайти в папку нашего проекта и выполнить команду:  

``` Brash
git init
```

Итак репозиторий создан, но пока контроль файлов отсутствует. Для того чтобы начать управлять версиями существующих файлов, укажите файлы за которыми должна следить система. Для этого понадобится несколько команд. (см. п.№5 Запись изменений в репозиторий).

## 5. Запись изменений в репозиторий.

После того как репозиторий был создан мы можем проверить статус(т.е. выяснить что GIT "думает" об этой папке). Для этого необходимо в терминале набрать команду git status. GIT выдаст сообщение :

``` Bash
On branch master

No commits yet

nothing to commit (create/copy files and use "git 

add" to track)
```

Чтобы начать отслеживать файлы в созданном репозитории GIT дает нам подсказку (```create/copy 
files and use "git add" to track```) необходимо создать файл в папке нажав на иконку New file, дать файлу название и указать расширение.(file.md как 
пример) Если проверить статус сразу после создания файла программа выдаст:
```Bash
No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        
        GIT ISTRUCTION VER 1.md

nothing added to commit but untracked files present 

(use "git add" to track)
```

Это означает что программа видит новый файл,но пока еще за ним не следит.

Для того чтоб программа начала отслеживать файл необходимо воспользоваться командой git add НАЗВАНИЕ файла (file.md)
```Brash
git add file.md
```
Теперь GIT начал отслеживать данный файл.

 ## 6. Просмотр истории изменений (коммитов).

Для просмотра все выполненных фиксаций можно воспользоваться историей коммитов. Она содержит сведения о каждом проведенном коммите проекта. 

Запросить ее можно при помощи команды:

``` Brash
git log
```

В ней содержится вся информация о каждом отдельном коммите, с указанием его хэша, автора, списка изменений и даты, когда они были сделаны.

## 8. Игнорирование файлов.

Для того чтобы исключить из отслеживания в репозитории определенные файлы или папки необходимо создать файл ***.gitignore*** и записать в него их названия или шаблоны соответсвующие таким файлам или папкам. 

Есть возможность игнорировать определенные файлы которые начинаются с определенного слова. Для этого необходимо указать имя, а затем *:
```Bash
img*
```
Эта команда позволит игнорировать все файлы и каталоги, имя которых начинается с ```img```.

Также можно создать шаблон для игнорирования файлов с определенным разрешением. Для этого вам необходимо будет использовать *, за которым последует расширение файла.

Например, если вы хотите игнорировать все файлы разметки, которые заканчиваются расширением .md, то вы должны в файл .gitignore добавить следующее:
```Bash
*.md
```
Этот шаблон будет соответствовать любому файлу с расширением .md, расположенному в любой папке проекта.

Чтобы сделать исключение для одного из файлов вам нужно будет воспользоваться шаблоном с отрицанием (с восклицательным знаком), чтобы исключить файл, который в противном случае был бы проигнорирован, как и все остальные:

!README.md  

 ```игнорирует все файлы .md```

``` не игнорирует файл README.md```

 Учитывая эти два шаблона в файле .gitignore, все файлы, оканчивающиеся на .md будут игнорироваться, кроме файла README.md.

## 10. Слияние веток и разрешение конфликтов.

Для слияния новой ветки с текущей необходимо ввести команду 

```Bash
git merge <название ветки которую хотим добавить в текущую>
```
Если была изменена одна и та же часть файла в обеих ветках, то может возникнуть конфликт,который может потребовать участие пользователя.

GIT предложит пользователю выбрать один из вариантов разрешения конфликта:

![screenconflict](screenconflicts.jpg)

Исходя из данного изображения видно, что пользователь может выбрать между несколькими вариантами:

```Accept current change``` (оставить информацию которая зафиксирована на текущей ветке)

```Accept incomming change```(заменить текущую информацию на информацию из сливаемой ветки)

```Accept both changes```(принять оба изменения)

```Compare changes```(сравнить изменения)

После того как конфликт был успешно разрешен,необходимо добавить коммит:
```Bash
git commit -m "Resolved merge conflict"
```
## 11. Удаление веток.

После того, как успешно проведено слияние веток, ненужные ветки могут быть удалены. Для этого необходимо воспользоваться коммандой:

```Bash
git branch -d <название ветки,кторая подлежит удалению>
```
 Перед удалением необходимо убедиться, что Вы не находитесь в момент удаления в ветке, которую пытаетесь удалить, программа выдаст ошибку:

![errorbranchdelete](git_2-1536x91.png)

Также ошибка может возникнуть когда Вы попытаетесь удалить ветку,изменения в которой не были зафиксированы(слиты),таким образом GIT оберегает те изменения, которые отлеживаются только на ветке,которую Вы пытаетесь удалить.

![errorbranchdelete](git_5-1536x163.png)

В случае если Вы всё таки уверенны что ветку можно удалить, необходимо воспользоваться командой:

```Bash
git branch -D <название ветки,кторая подлежит удалению>
```
## 12. Работа с удаленными репозиториями.


### Работа со своими репозиториями.

  
  * Создать аккаунт на GitHub.

  Для этого необходимо ввести адрес Вашей электронной почты и название профиля(оно будет использоваться в интерфейсе, в коммитах и комментариях.) Если имя недоступно, GitHub вам об этом скажет. А если доступно — жмите Continue.

  Теперь у вас есть аккаунт. Можете создать репозиторий и работать с ним, или склонировать чужой.

* Создать локальный репозиторий. 

Создание локального репозитория уже рассматривалось в Разделе **4. Инициализация репозитория**, и не должно вызвть затруднений.

* Создать удаленный репозиторий. 

Вы создали локальный репозиторий, теперь, например, вам нужно добавить его на Github, тем самым вы фактически создадите удаленный репозиторий.

Перейдите на https://github.com и войдите в свой аккаунт. Нажмите кнопку New repository (Новый репозиторий). На открывшейся странице введите имя репозитория (Repository name) и нажмите кнопку Create repository.

* Связать удаленный репозиторий с локальным.

В своем локальном репозитории теперь выполните ***3 команды***:
```Bash
git remote add origin https://github.com/username/myproject.git
git branch -M main
git push -u origin main
```
Для получения и слияния изменений из удаленного репозитория используется команда ```git pull```.

Отправить изменения локального репозитория в удаленный можно с помощью команды ```git push```.

### Работа с чужими репозиториями.

Через GitHub есть возможность также работать с чужими репозиториями и принимать участие в проектах других людей. Для поиска нужных проектов можно воспользоваться поисковой строкой. Всю информацию можно посмотреть как на странице GitHub,а также можно загрузить проект на свой локальный компьютер для этого необходимо воспользоваться кнопкой CODE>Local>скопировать HTTPS адрес. Затем с помощью команды git clone (HTTPS адрес необходимого репозитория). Теперь мы имеем доступ к удаленному чужому проекту. Мы можем посмотреть всю историю коммитов,слияний, количество и имена учстников и другую информацию. 

Для того чтобы внести какие-то изменения в чужой репозиторий(проект) необходимо сначала сделать его копию. Это возможно сделать с помощью кнопки **```Fork```**. После создания копии мы автоматически оказываемся на странице скопированного проекта. Технически теперь это ваш репозиторий, но скопированный с чужого. Для того чтобы вносить какие-то изменения в репозитории необходимо создать новую ветку, в которой будет продолжаться работа до того момента, пока ваши изменения не будут приняты владельцем скопированного репозитория. После того как вы закончли работу, необходимо всю информацию направить на удаленный репозиторий. Так как наша ветка еще никак не связана с удаленным репозиторием, нам поможет команда
```Bash
git push -u origin new
```

Теперь наша ветка отображается в удаленном репозитории и для того чтобы нам предложить свои изменения воспользуемся кнопкой PullRequest. После нажатия появятся предложенные изменения, нажимаем на кнопку Create PullRequest, заполняем заголовок(то что увидит автор)добавляем комментарии к нашим изменениям, и отправляем изменения.
