## 4. Инициализация репозитория
**File-open folder** после чего выбрать нужную папку. 

После выбора папки в левой рабочей области созадём файл, в котором будем работать.
Например: *GitInstruct.md* Для того, чтобы инициализировать локальный репозиторий необходимо ввести команду `git init`. 

После инициализации с помощью команды `git status` нужно проверить всё ли сработало, если да, то ничего не изменится, и будет показан текущий статус. 
## 5. Запись изменений в репозиторий
После внесения каких-либо изменений в файл, необходимо добавить файлы к коммиту с помощью следующих команд:
`git add <имя файла>`  (P.S.Чтобы полностью не записывать имя файла можно ввести первые несколько символов от его имени, а дальее нажать *"Tab"* и терминал самостоятельно подставит до конца)
После того, как мы добавили файлы, необходимо закоммитить изменения, чтобы не потерять при помощи команды:
`git commit -m "Сообщение о коммите"`
Можно эти 2 команды выполнить вместе за один раз,для этого необходимо написать следующую команду: `git commit -am "Сообщение о коммите"`
## 6. Просмотр истории коммитов
Для того, чтобы просмотреть историю всех коммитов необходимо написать команду: `git log`. Терминал выведет нам всю историю и их хэш-коды(**Например, commit 55b68f39a9ca24166671c7cebef699431594a01b** с их помощью можно перемещаться между сохранениями.)
Так же можно использвать команду: `git log --graph`, при помощи которой можно увидеть в каких ветках производилась работа
 Для того, чтобы изменить сообщение последнего коммита, необходимоввести команду: `git commit --amed`.  Эта команда откроет в вашем текстовом редакторе сообщение вашего последнего коммита, для того, чтобы вы могли его исправить.
 Так же можно посмотреть историю всех коммитов в более сжатом варианте, в 1 строку, для этого необходимо ввести команду `git log --oneline --all`

## 8. Игнорирование файлов
Для того, чтобы исключить из oтслеживания в репозитории определенного файла или папки необходимо там создать файл ***.gitignore*** и записать в него их названия или шаблоны, соответствующие таким файлам или папкам.
В файл **.gitignore** должны быть добавлены файлы любого типа, которые не нужно фиксировать.

Если вы хотите игнорировать все файлы с определенным расширением, то вам необходимо будет использовать селектор подстановочного знака *, за которым последует расширение файла.

Например, если вы хотите игнорировать все файлы разметки, которые заканчиваются расширением .md, то вы должны в файл .gitignore добавить следующее:
***\*.md***. Этот шаблон будет соответствовать любому файлу с расширением .md, расположенному в любой папке проекта, он позволит игнорировать все файлы, оканчивающиеся на .md, но если вы, например, не хотите, чтобы Git игнорировал файл README.md.
Для этого нужно будет воспользоваться шаблоном с отрицанием (с восклицательным знаком), чтобы исключить файл, который в противном случае был бы проигнорирован, как и все остальные:
***\.md*** -  игонорирует все файлы с расширением .md
***!Readme*** - Git не игнорирует файл Readme
## 9. Создание веток в Git
По умолчанию имя основной ветки Git *master/main*
Создать ветку можно командой:

`git branch <Имя новой ветки>`
.

Список веток в репозитории можно посмотреть с помощью команды `git brach`. Текущая ветка будет отмечена звездочкой **main**
Для переклчения между ветками используется команда `git checkout` или `git switch`

 `git chekout -b` - создание новой и сразу перейти на нёё

 Для того, чтобы посмотреть последний сохранённый коммит в каждой ветке можно использовать команду:
 `git branch -v`
 
 Для того, чтобы посмотреть предыдущий коммит в ветке можно воспользоваться командой `git checkout branch^2`

## 10. Слияние веток и разрешение конфликтов
Для того чтобы выполнить слияние 2-х веток, необходимо воспользоваться командой:
`git merge <branch name>`

Так же можно выполнять объединенте при помощи команды `git rebase` Смысл этой команды в том, что тут происходит наложение коммитов поверх другого базового коммита
![Logo](rebase.jpg)

**Необходимо учитывать, что при сляинии происходит присоединение той ветки, которую указал в команде, в ту, в которой находишься**
 При слиянии веток могут произойти *Конфликты слияния* Они происходят при слиянии ветвей, имеющих конкурирующие фиксации, и Git требуется  помощь, чтобы принять решение относительно того, какие изменения следует включить в окончательное слияние.
 Выглядит конфликт, как правило, следующим образом: 
 ![Logo](конфликт.jpg)

 ![Logo](конфликтт.jpg)

 Для того, чтобы разрешить конфликт существует несколько вариантов:
 1. Внести изменения в одну из веток, в строках, в которых произошёл конфликт
 2. Выбрать один из предложенных вариантов разрешения в VS Code: **Accept Current Change\Accept Incoming Change\Acept Both Change**
## 11. Удаление ветки
С операцией удаления над ветками справляется   команда `git branch  -d`

Для корректного удаления нужно помнить несколько правил, чтобы не получить ошибки:

1. Нельзя удалить ветку, в которой вы находитесь. Git выкинет ошибку и не произведет удаление. Следовательно, нужно перейти на другую ветку.

2. Git не позволит удалить ветку, у которой есть несохраненные изменения. Так мы избегаем ситуации, когда часть написанного кода будет безвозвратно утеряна. Если же мы уверены, что изменения в этой версии не нужны и их можно смело удалять, то вместо флага -d используем -D:*

## 13 Работа с удалёнными репозиториями (remote repo)
Для того, чтобы установить связь локального репозитория с удалённым необходимо использовать команду `git remote add origin <url repo>`

Чтобы проверить привязку, можно выполнить команду `git remote` или `git remote -v`

Если текущая ветка в локальном репо отличается (именем) от удалённой, необходимо ее переименовать при помощи команды `git branch -M main`

После выполнения всех действий нужно отправить изменения в удалёный репо при помощи команды `git push -u origin main`

 ***При первой загрузке может вылезти ошибка, это связано с тем, что новая локальная ветка не имеет аналога в удалённом репо. Для того чтобы это исправить можно воспользоваться кмандой, которую предложит git
![Logo](ошибка.jpg)***

 Для того,чтобы скачать данные из удалённого репозитория в локальный можно использовать команду `git fetch` - эта команда только скачивает все необходимые дынные, но не сливает их с уже имеющимися в локальном репо.


Для того, чтобы загрузить изменения, произведенные в удалённом репо в локальный,необходимо воспользоваться командой `git pull`

При слиянии изменений из разных репозиториев могут возникнуть конфликты, аналогичные тем, что воникают при слиянии веток. Разрешается тоже, аналогично тому, как решается с конфликтами в ветках, если добавляем в локальный.
 Если же, пробуем запушить свои изменения, то в терминале может появиться следующая ошибка: 
 ![Logo](mistake.jpg)

 Для рещения этой проблемы необходимо сначала обновить свой репозиторий `pull`, а после запушить

 Для того, чтобы получить чужой удалённый репозиторий себе на локальный для работы с ним,можно его склонировать при помощи команды:
 `git clone <ссылка на проект>`
